Overview

shikken is a library for the R programming language that provides an interface to the shogun-toolbox, a C++ library for "large scale" machine learning. While the shogun-toolbox comes with its own R wrappers, this library is written in a manner that's more idiomatic to R.

For instance, building a 2-class SVM for the iris dataset works like so: 

  > library(shikken)
  > iris.sub <- subset(iris, Species != "virginica")
  > svm <- SVM(Species ~ ., data=iris.sub)
  > table(predict(svm, type='decision'), iris.sub$Species)
  
          setosa versicolor virginica
    -1         0         50         0
     1        50          0         0
(note that the labels are turned to -1/1 internally and are shown as such in the left column)

This library is in its early stages of development (started ~ June 20, 2011), and isn't quite ready for public consumption just yet. You can find more information on its wiki pages.

Why?

The static interface to the shogun-toolbox is (i) a bit alien to program against; (ii) doesn't expose all shogun's the functionality; and (iii) only allows you to work with one machine in your workspace at a time.

The R modular interface is currently at alpha quality and relies on SWIG, which still has a few problems working with R.

These things aside, both the static and modular interfaces still let you use lots of shogun functionality already, which is still great, so ... really ... why?

Being that I do much of my work in R, I thought I could write an interface that felt a bit more natural to the "R native," which is really only possible to if you write the library by hand. Also, I think I can provide a "mostly sensible" way of keeping up with shogun's garbage collection scheme, such that users of this library can largely avoid gradual memory leaks so long as they access the objects exposed herein via their normal R getter/setter type of generic functions.

I also like how the maintainers of the shogun-toolbox foster its open source nature by engaging with the user community, and I thought this would be a way to give back and help provide something useful for the community at large.

But really I needed to do some stuff with string kernels and thought "hey, why don't I try to do this real quick ... it shouldn't be too hard to wrap this all up in a nicer looking library, right?" Riiiiggghht ...
